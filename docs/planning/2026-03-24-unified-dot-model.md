# 実装計画: 統一ドットモデル

## 現状の問題

歴史都市ドットの明るさ・数・位置が別々のルールで決まっており、一貫性がない。
- Chandler データに西洋バイアス（地中海に都市が多い）
- scatter ドットの数が国面積に引きずられる
- 明るさを国人口にしたら国内の僻地も大都市と同じ明るさ
- 補正パッチを重ねて複雑化

## 統一ルール

**1ドット = N人**。これだけ。

### 古代〜近代 (〜1800)
- 各国の人口を N で割った数だけドットを配置
- ドットの位置: 既知都市（Chandler/手動）の周囲に人口比で配分。都市データがない国は国内のドット位置（NASA由来）にランダム配分
- ドットの明るさ/サイズ: **全ドット同一**（数で人口を表現する）
- N の値: 全世界のドット総数が見やすい範囲になるように調整。目安 50,000人/ドット

### 過渡期 (1800〜1950)
- 古代モデル（人口比ドット数）→ 近現代モデル（NASAドット）にフェードで遷移
- 古代ドットのα を下げながら NASA ドットのα を上げる

### 近現代 (1950〜2100)
- 現行の NASA ドット + コホート年齢色変え（変更なし）
- 成長国のドット増殖（現行の growth slot 方式）

## データフロー

### Python 事前計算（historical_dots.json を生成）
```
入力:
  - pop_history.json（国別年別人口）
  - historical_cities.json（都市位置 — 配置の参考のみ）
  - city_dots.json（国内のドット位置候補 — scatter 用）

処理:
  for each timeline year <= 1800:
    for each country:
      n_dots = floor(national_pop / N)
      positions = allocate_to_cities_and_scatter(n_dots, known_cities, country_dots)
      emit dots

出力:
  historical_dots.json = {
    year_string: [
      {lat, lon},  // 全ドット同一サイズ・色なので属性不要
      ...
    ],
    ...
  }
```

### JS 側
- `historical_dots.json` をロード
- 年が変わるたび、その年のドット配列を取得
- THREE.Points のバッファに書き込み（位置のみ更新、色/サイズは era に応じて固定）
- バッファサイズ: 全年の最大ドット数で確保

## N の決定

| N (人/dot) | BC2000 世界(63M) | AD1 世界(223M) | AD1000 世界(323M) |
|-----------|-----------------|---------------|-----------------|
| 10,000 | 6,300 dots | 22,300 dots | 32,300 dots |
| 50,000 | 1,260 dots | 4,460 dots | 6,460 dots |
| 100,000 | 630 dots | 2,230 dots | 3,230 dots |

50,000 が妥当。BC2000 で 1,260ドット（sparse だが見える）、AD1000 で 6,460ドット（dense すぎない）。

## ドット配置アルゴリズム

国に n_dots を割り当てるとき:
1. その国の既知都市（Chandler + 手動）の人口を集計
2. 各都市に `floor(city_pop / national_pop * n_dots)` を割り当て
3. 残りは都市周辺にランダム散布（seeded RNG）
4. 都市がない国は、NASA ドット位置から均等に選択

都市の周囲に配置 = 都市を中心にガウシアン分布（σ = 都市人口の sqrt に比例）

## 色・サイズ

| 時代 | 色 | サイズ | alpha |
|------|-----|--------|-------|
| 古代 (era=0) | 赤い火 (R=1.0 G=0.3 B=0.05) | 2.5 (全ドット同一) | 0.8 |
| 過渡期 (0<era<1) | 火→電気にブレンド | 2.5→origSizes にブレンド | ブレンド |
| 近現代 (era=1) | 年齢色変え | origSizes | ratio ベース |

## 影響範囲

- **新規**: `generate_historical_dots.py` — 事前計算スクリプト
- **新規**: `historical_dots.json` — 年別ドット位置データ
- **変更**: `index.html` — 歴史ドット描画ロジックを簡素化

## 実装フェーズ

### Phase 1: Python で historical_dots.json 生成
- N = 50,000 で全 timeline year のドット配置を計算
- 検証: BC2000/BC1000 で中国のドット数が人口比に正確か確認

### Phase 2: JS 側書き換え
- historical_dots.json をロード
- 歴史ドットの THREE.Points を年ごとのドット配列で更新
- 位置のみ更新、色/サイズは era で固定

### Phase 3: 過渡期のブレンド
- 1800〜1950 で歴史ドット（フェードアウト）と NASA ドット（フェードイン）を重ねる

## リスク

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| historical_dots.json のサイズ | MEDIUM | 年数(~200) × ドット数(~6000) × 8byte ≈ 10MB。gzip で 1-2MB |
| 年ごとにバッファ全書き換えのパフォーマンス | LOW | 6000ドットのFloat32Array コピーは <1ms |
| 古代→近代の視覚的な不連続 | MEDIUM | 過渡期のフェードで緩和 |
| N=50,000 が大きすぎ/小さすぎ | LOW | 試して調整 |

## 複雑度: 中

これまでのパッチ群を捨てて統一ルールに置き換えるので、コード量は減る。
